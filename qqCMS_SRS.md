## 1. Введение

### 1.1 Назначение документа

Настоящий документ является техническим заданием (Software Requirements Specification, SRS) на разработку системы управления контентом **qqCMS**. Документ составлен в соответствии с международным стандартом ISO/IEC/IEEE 29148:2018 «Systems and software engineering — Life cycle processes — Requirements engineering».

Документ предназначен для:
- разработчиков системы (Артур Паничев, Денис Дунаев);
- руководителя дипломного проекта (Сергей Лобышев);
- оценочной комиссии IVKHK.

### 1.2 Область применения

**qqCMS** — это веб-ориентированная система управления контентом с открытым исходным кодом, предназначенная для создания, редактирования и публикации многоязычных веб-сайтов. Система реализована в виде монорепозитория, включающего три независимых приложения: публичный клиент, административная панель и серверная часть.

Система разрабатывается как дипломный проект в рамках учебной программы IVKHK.

## 2. Общее описание

### 2.1 Контекст и назначение системы

На рынке существуют как тяжёлые enterprise-CMS (WordPress, Drupal), так и headless-CMS (Strapi, Contentful), однако большинство из них либо избыточны по функциональности, либо требуют платной подписки, либо не предоставляют типобезопасного API «из коробки».

**qqCMS** разрабатывается как легковесная, типобезопасная и многоязычная CMS с современным стеком технологий (Bun, Astro, Elysia, Drizzle ORM), пригодная для развёртывания на собственной инфраструктуре.

### 2.2 Пользователи системы

| Роль | Описание |
|---|---|
| Администратор сайта | Единственный пользователь административной панели. Управляет контентом, навигацией, языками и настройками сайта. Учётные данные задаются через переменные окружения (`.env`). |
| Посетитель сайта | Анонимный пользователь, просматривающий опубликованный контент через публичный клиент. |

### 2.3 Допущения и зависимости

- Система разрабатывается для развёртывания на Linux-сервере.
- По умолчанию используется PGLite (PostgreSQL в WASM, без сервера); при необходимости возможно переключение на PostgreSQL через `.env`.
- Медиафайлы хранятся в локальной файловой системе сервера.
- Функция регистрации пользователей и управления ролями не входит в текущую версию.
- Кастомные блоки контента (hero, карусели и т.д.) проектируются архитектурно, но не реализуются в текущей версии.

### 2.4 Архитектурный обзор

Система состоит из трёх приложений в едином монорепозитории (Bun Workspaces) и одного общего пакета:

```
qqcms/
├── apps/
│   ├── client/          # Публичный сайт (Astro SSG)
│   ├── admin/           # Административная панель (Astro + Preact Islands)
│   └── server/          # Backend API (Elysia + Drizzle ORM)
└── packages/
    └── shared/          # Общие типы, схемы, Treaty-клиент, i18n
```

## 3. Требования к системе (SyRS)

### 3.1 Системные функциональные требования

**SYS-F-01** Система должна обеспечивать создание, редактирование и удаление веб-страниц.

**SYS-F-02** Система должна поддерживать многоязычный контент: администратор добавляет поддерживаемые языки сайта, и каждая страница должна быть заполнена на каждом из добавленных языков.

**SYS-F-03** Система должна обеспечивать управление навигацией сайта (ссылки в шапке).

**SYS-F-04** Система должна позволять назначать любую страницу главной.

**SYS-F-05** Система должна обеспечивать загрузку и хранение медиафайлов (изображения).

**SYS-F-06** Система должна публиковать контент в виде статически сгенерированного сайта.

**SYS-F-07** Система должна обеспечивать авторизацию администратора по логину и паролю из `.env`.

**SYS-F-08** Публичный клиент должен отображать выбор языка на основании языков, настроенных в административной панели.

### 3.2 Системные нефункциональные требования

**SYS-NF-01** Система должна работать на Bun runtime версии 1.3.10 и выше.

**SYS-NF-02** Система должна поддерживать PGLite (по умолчанию, без сервера) и PostgreSQL (через `.env`).

**SYS-NF-03** Система должна быть пригодна для развёртывания посредством Docker.

**SYS-NF-04** Все взаимодействия между фронтендом и бэкендом должны быть типобезопасными (через Elysia Treaty).

## 4. Требования к программному обеспечению (SRS)

### 4.1 Серверная часть (apps/server)

#### 4.1.1 Технологический стек

| Компонент | Технология |
|---|---|
| Runtime / Package Manager | Bun |
| HTTP-фреймворк | Elysia |
| ORM | Drizzle ORM |
| База данных (основная) | PGLite (PostgreSQL/WASM) |
| База данных (альтернативная) | PostgreSQL |
| Валидация | Zod |
| Аутентификация | JWT (httpOnly cookie) |
| Линтер / Форматтер | Biome |

#### 4.1.2 Архитектура

Бэкенд реализуется по **модульной архитектуре**: каждый домен (страницы, навигация, медиа, настройки, языки, авторизация) оформляется в виде отдельного модуля со своими контроллером, сервисом и схемой.

```
src/
├── modules/
│   ├── auth/
│   ├── pages/
│   ├── navigation/
│   ├── media/
│   ├── settings/
│   └── languages/
├── db/
│   ├── index.ts
│   ├── schema/
│   └── migrations/
├── middleware/
│   ├── auth.middleware.ts
│   ├── cors.middleware.ts
│   └── rateLimit.middleware.ts
├── lib/
│   ├── jwt.ts
│   ├── password.ts
│   └── env.ts
├── app.ts
└── index.ts
```

#### 4.1.3 Функциональные требования — сервер

**SRV-F-01** Сервер должен предоставлять типобезопасный API через Elysia Treaty.

**SRV-F-02** Сервер должен реализовывать CRUD-операции для сущностей: страницы (pages), переводы страниц (page_translations), навигация (navigation_items), медиа (media), настройки (settings), языки (languages).

**SRV-F-03** Сервер должен принимать и хранить медиафайлы в директории `uploads/` локальной файловой системы.

**SRV-F-04** Сервер должен проверять JWT-токен для всех защищённых маршрутов.

**SRV-F-05** Сервер должен отдавать webhook-уведомление для триггера ребилда статического клиента при сохранении контента.

**SRV-F-06** Сервер должен поддерживать CORS с настраиваемым списком разрешённых источников через `.env`.

**SRV-F-07** Сервер должен применять rate limiting для API-эндпоинтов.

**SRV-F-08** Переключение между PGLite и PostgreSQL должно осуществляться исключительно через переменную окружения `DATABASE_URL` без изменения кода.

#### 4.1.4 Схема базы данных

| Таблица | Описание |
|---|---|
| `pages` | Страницы сайта: slug, is_homepage, created_at, updated_at |
| `page_translations` | Переводы страницы: page_id (FK), language_code, title, content (JSON TipTap) |
| `navigation_items` | Пункты навигации: label (JSON i18n), href, order, parent_id (self-ref) |
| `media` | Медиафайлы: filename, original_name, mime_type, size, url |
| `languages` | Поддерживаемые языки сайта: code (en/ru/et), label, is_active |
| `settings` | Настройки ключ-значение: key, value (JSON) |

Первичные ключи — UUID v4. Миграции управляются через Drizzle Kit.

### 4.2 Административная панель (apps/admin)

#### 4.2.1 Технологический стек

| Компонент | Технология |
|---|---|
| Фреймворк | Astro (Islands Architecture) |
| UI-компоненты | Preact (с поддержкой React-библиотек через `@preact/compat`) |
| UI-библиотека | DaisyUI (Tailwind CSS плагин, нативная совместимость с Preact) |
| Стейт | Preact Signals (собственные сторы) |
| Стилизация | Tailwind CSS |
| Редактор контента | TipTap |
| Линтер / Форматтер | Biome |
| Методология компонентов | Atomic Design |

#### 4.2.2 Структура компонентов (Atomic Design)

```
src/
├── components/
│   ├── atoms/          # Button, Input, Label, Badge, Spinner
│   ├── molecules/      # FormField, LanguageTabs, NavMenuItem
│   ├── organisms/      # PageEditor, NavigationEditor, MediaUploader, LoginForm
│   └── templates/      # AdminLayout
├── pages/
├── islands/            # Интерактивные Preact-острова
├── stores/             # Preact Signals сторы
└── lib/
```

#### 4.2.3 Функциональные требования — административная панель

**ADM-F-01** Панель должна отображать форму авторизации (логин + пароль) и выполнять вход через API, сохраняя JWT в httpOnly cookie.

**ADM-F-02** Панель должна предоставлять список всех страниц с возможностью создания, редактирования и удаления.

**ADM-F-03** При редактировании страницы должны отображаться вкладки для переключения между языками сайта; каждая вкладка содержит поля title и content (TipTap редактор).

**ADM-F-04** Редактор TipTap должен поддерживать:
- форматирование текста (жирный, курсив, заголовки, списки, ссылки);
- вставку изображений из медиатеки;
- вставку YouTube-видео по URL;
- расширяемую архитектуру для подключения кастомных блоков в будущем (без реализации самих блоков).

**ADM-F-05** Панель должна предоставлять интерфейс управления навигационными ссылками шапки сайта (добавление, редактирование, удаление, изменение порядка).

**ADM-F-06** Панель должна позволять выбирать, какая страница является главной (homepage).

**ADM-F-07** Панель должна предоставлять раздел управления языками сайта: добавление/удаление языков (EN, RU, ET и др.).

**ADM-F-08** Панель должна предоставлять медиабиблиотеку с загрузкой файлов и просмотром загруженных изображений.

**ADM-F-09** Интерфейс административной панели должен быть переведён на русский и английский языки; язык интерфейса переключается пользователем.

**ADM-F-10** После сохранения контента панель должна инициировать ребилд публичного клиента (через webhook).

### 4.3 Публичный клиент (apps/client)

#### 4.3.1 Технологический стек

| Компонент | Технология |
|---|---|
| Фреймворк | Astro (SSG) |
| UI-компоненты | Preact |
| Стилизация | Tailwind CSS |
| Линтер / Форматтер | Biome |

#### 4.3.2 Функциональные требования — публичный клиент

**CLI-F-01** Клиент генерируется статически (SSG) на основе данных API во время сборки.

**CLI-F-02** Клиент должен поддерживать многоязычную маршрутизацию вида `/{lang}/{slug}`, где `{lang}` — один из активных языков сайта.

**CLI-F-03** При открытии корневого URL `/` клиент должен автоматически определять предпочтительный язык браузера и выполнять редирект на соответствующий языковой раздел. При отсутствии совпадения — редирект на язык по умолчанию.

**CLI-F-04** Клиент должен отображать шапку с навигационными ссылками, настроенными в административной панели.

**CLI-F-05** В шапке должен быть переключатель языков, отображающий только те языки, которые активированы в административной панели.

**CLI-F-06** Клиент должен корректно рендерить HTML-контент, сгенерированный TipTap (включая изображения и встроенные YouTube-видео).

**CLI-F-07** Клиент должен отображать главную страницу, назначенную в административной панели.

## 5. Нефункциональные требования

### 5.1 Производительность

**NFR-PERF-01** Публичный клиент (SSG) должен достигать оценки Lighthouse Performance ≥ 90 на десктопе.

**NFR-PERF-02** API-эндпоинты должны обрабатывать запросы за ≤ 200 мс при нагрузке до 50 одновременных запросов (локальная PGLite).

### 5.2 Безопасность

**NFR-SEC-01** Пароль администратора не должен храниться в базе данных; используется сравнение с хешированным значением из `.env`.

**NFR-SEC-02** JWT-токен должен храниться в httpOnly cookie и не должен быть доступен через JavaScript на клиенте.

**NFR-SEC-03** Все защищённые маршруты API должны проверять JWT перед обработкой запроса.

**NFR-SEC-04** Загружаемые файлы должны валидироваться по MIME-типу и ограничиваться допустимыми форматами изображений.

**NFR-SEC-05** Система должна применять заголовки безопасности HTTP (Content-Security-Policy, X-Frame-Options, X-Content-Type-Options).

### 5.3 Удобство сопровождения

**NFR-MAIN-01** Весь код должен соответствовать правилам Biome (lint + format) и проходить проверку в CI.

**NFR-MAIN-02** Схема базы данных должна управляться через Drizzle Kit миграции; изменения схемы не должны требовать ручного SQL.

**NFR-MAIN-03** Монорепозиторий должен быть организован через Bun Workspaces с чёткими зависимостями между пакетами.

**NFR-MAIN-04** Каждый модуль бэкенда должен быть самодостаточным (контроллер + сервис + схема) и не иметь circular dependencies.

### 5.4 Переносимость и развёртывание

**NFR-PORT-01** Каждое из трёх приложений должно быть упаковано в Docker-образ.

**NFR-PORT-02** Все параметры конфигурации (DATABASE_URL, ADMIN_LOGIN, ADMIN_PASSWORD, JWT_SECRET, CORS_ORIGINS) должны задаваться исключительно через переменные окружения.

**NFR-PORT-03** Система должна поддерживать развёртывание на Render.com и аналогичных платформах.

### 5.5 Масштабируемость

**NFR-SCALE-01** Переключение с PGLite на PostgreSQL должно осуществляться без изменения кода приложения.

**NFR-SCALE-02** Архитектура редактора TipTap должна допускать подключение кастомных блоков контента (hero, карусели и т.д.) в последующих версиях без рефакторинга ядра редактора.

### 5.6 Интернационализация

**NFR-I18N-01** Система должна поддерживать произвольный набор языков сайта, добавляемых через административную панель (не ограничиваясь EN/RU/ET).

**NFR-I18N-02** Интерфейс административной панели должен быть доступен на русском и английском языках.

## 6. Ограничения и допущения

### 6.1 Ограничения

**CON-01** В текущей версии поддерживается только одна роль пользователя — администратор.

**CON-02** Кастомные блоки TipTap (hero, карусели, аккордеоны и т.д.) архитектурно предусмотрены, но не реализуются в данной версии.

**CON-03** Медиафайлы хранятся только на локальной файловой системе; интеграция с S3 или иными облачными хранилищами не входит в текущую версию.

**CON-04** Публичный клиент является статически сгенерированным (SSG); обновление контента требует ребилда.

**CON-05** Версионирование API не предусмотрено в текущей версии.

**CON-06** Регистрация новых пользователей не предусмотрена; учётные данные администратора задаются в `.env`.

### 6.2 Допущения

**ASM-01** Разработка ведётся на Windows (WSL2 / Bun).

**ASM-02** Целевая производственная среда — Linux-сервер с Docker.

**ASM-03** Проект использует единый Git-репозиторий (монорепо).

**ASM-04** Исходный код размещается в публичном или приватном репозитории на GitHub.

## 7. Требования к верификации и валидации

### 7.1 Критерии приёмки

| ID требования | Метод верификации | Критерий успеха |
|---|---|---|
| SRV-F-01 | Интеграционное тестирование | Treaty-клиент успешно вызывает все API-эндпоинты с типизированными параметрами |
| SRV-F-02 | Функциональное тестирование | CRUD-операции для всех сущностей выполняются без ошибок |
| SRV-F-04 | Тестирование безопасности | Запрос к защищённому эндпоинту без токена возвращает HTTP 401 |

### 7.2 Тестовые сценарии (высокого уровня)

**TS-01 — Полный цикл управления страницей**
1. Администратор входит в систему.
2. Создаёт страницу с slug `about`, заполняет title и content на EN и RU.
3. Назначает страницу главной.
4. Запускает ребилд клиента.
5. Публичный клиент по URL `/en/about` отображает английскую версию, по `/ru/about` — русскую.

**TS-02 — Управление языками**
1. Администратор добавляет язык `ET` (эстонский).
2. На всех существующих страницах появляется вкладка ET.
3. Публичный клиент отображает переключатель с тремя языками.

**TS-03 — Загрузка и вставка медиа**
1. Администратор загружает изображение через медиабиблиотеку.
2. В редакторе TipTap вставляет это изображение на страницу.
3. После ребилда изображение отображается на публичном клиенте.

**TS-04 — Вход в систему**
1. Попытка входа с неверным паролем — система возвращает ошибку авторизации.
2. Вход с корректными данными — выдаётся JWT в httpOnly cookie, открывается дашборд.
