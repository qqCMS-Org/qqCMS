# Public Web Client (`apps/web`)

> **Status: IMPLEMENTED** — Phase 7 (Steps 1–6) is complete. See `docs/TODO.md` for the full checklist.

## Overview

`apps/web` is the public-facing website generated by qqCMS. It is built with Astro in **static** mode (Astro v5+):

- Content pages (`/[lang]/[...slug]`) are **prerendered at build time** (SSG) — every published page × every active language becomes a static HTML file.
- The root `/`, `/api/revalidate`, and `404.astro` are **SSR** — they need request-time data (`Accept-Language` header, deploy hook) or cannot be prerendered.

Architecture mirrors `apps/admin` exactly: same FSD layers, same path aliases, same Eden Treaty client pattern, same Tailwind + DaisyUI setup.

## Stack

| Component | Technology |
|---|---|
| Framework | Astro (hybrid mode) |
| UI islands | Preact (same as admin) |
| Shared primitives + TipTap renderer | `@repo/ui` |
| Styling | Tailwind CSS + DaisyUI (same theme as admin) |
| API client | Eden Treaty (`@repo/server/client`) |

## Content rendering lives in `packages/ui`

TipTap node components are **Preact TSX** in `packages/ui/src/tiptap/`. Preact components imported in Astro pages without a `client:*` directive render as static HTML — zero JS shipped to the browser.

This means the same components can later be used in `apps/admin` for a live page preview, without any duplication.

```
packages/ui/src/tiptap/
├── types.ts               # TipTapNode interface, NodeRegistry type
├── registry.ts            # defaultRegistry: Record<nodeType, Component>
├── TipTapRenderer.tsx     # recursive renderer — walks node.content[], looks up registry
├── nodes/
```

## Adding a Custom Content Block

Adding a new block (e.g., "Testimonial") is handled entirely within `packages/ui`. The Admin panel automatically detects new blocks and enables inline editing.

**1. Create the block in `packages/ui/src/blocks/`**

```tsx
// packages/ui/src/blocks/TestimonialBlock.tsx
import { Editable } from "./Editable";
import type { BlockComponentProps, BlockDefinition } from "./types";

export interface TestimonialAttrs {
  quote: string;
  author: string;
}

function TestimonialComponent({ attrs, isEditing, updateAttrs }: BlockComponentProps<TestimonialAttrs>) {
  return (
    <div class="p-8 border rounded-xl">
      <Editable
        tag="blockquote"
        value={attrs.quote}
        onChange={(val) => updateAttrs?.({ quote: val })}
        isEditing={isEditing}
        className="text-xl italic"
        multiline
      />
      <Editable
        tag="cite"
        value={attrs.author}
        onChange={(val) => updateAttrs?.({ author: val })}
        isEditing={isEditing}
        className="block mt-4 font-bold"
      />
    </div>
  );
}

export const TestimonialBlock: BlockDefinition<TestimonialAttrs> = {
  type: "testimonial",
  label: "Testimonial",
  description: "Customer quote with author name",
  defaultAttrs: {
    quote: "This is an amazing product!",
    author: "John Doe",
  },
  Component: TestimonialComponent,
};
```

**2. Register it in `packages/ui/src/blocks/index.ts`**

```ts
import { HeroBlock } from "./HeroBlock";
import { TestimonialBlock } from "./TestimonialBlock"; // Import

export const blockRegistry: BlockRegistry = {
  hero: HeroBlock,
  testimonial: TestimonialBlock, // Add to registry
};
```

**3. Automatic Integration**

Once registered:
- The block appears in the Admin **Slash Menu** (`/`).
- **Inline Editing** is enabled for all `<Editable />` fields.
- **Technical Settings** (like `bgImage` or `buttonHref`) appear in a floating popover if defined in `defaultAttrs`.
- The block renders identically on the live site.

## Environment Variables

| Variable | Description | Required |
|---|---|---|
| `PUBLIC_API_URL` | URL of the API server | Yes |
| `REVALIDATE_SECRET` | Shared secret for the `/api/revalidate` webhook | Yes |
| `DEPLOY_HOOK_URL` | Render.com (or other) deploy hook to trigger rebuild | Production only |

## Rebuild Flow

1. Admin saves content → API calls `POST {PUBLIC_CLIENT_URL}/api/revalidate`
2. `revalidate.ts` verifies `x-revalidate-secret` with `crypto.timingSafeEqual`
3. Valid → calls `DEPLOY_HOOK_URL`; absent in dev → returns `{ ok: true }`
4. New build fetches fresh data → regenerates all static HTML
