# Public Web Client (`apps/web`)

> **Status: PLANNED** — See Phase 6–7 in `docs/TODO.md` for the implementation checklist.

## Overview

`apps/web` is the public-facing website generated by qqCMS. It is built with Astro in **hybrid** mode:

- Content pages (`/[lang]/[...slug]`) are **prerendered at build time** (SSG) — every published page × every active language becomes a static HTML file.
- The root `/` and `/api/revalidate` run as **SSR** — they need request-time data (`Accept-Language` header, deploy hook).

Architecture mirrors `apps/admin` exactly: same FSD layers, same path aliases, same Eden Treaty client pattern, same Tailwind + DaisyUI setup.

## Content rendering lives in `packages/ui`

TipTap node components are **Preact TSX** in `packages/ui/src/tiptap/`. Preact components imported in Astro pages without a `client:*` directive render as static HTML — zero JS shipped to the browser.

This means the same components can later be used in `apps/admin` for a live page preview, without any duplication.

```
packages/ui/src/tiptap/
├── types.ts               # TipTapNode interface, NodeRegistry type
├── registry.ts            # defaultRegistry: Record<nodeType, Component>
├── TipTapRenderer.tsx     # recursive renderer — walks node.content[], looks up registry
├── nodes/
│   ├── DocNode.tsx
│   ├── ParagraphNode.tsx
│   ├── HeadingNode.tsx
│   ├── TextNode.tsx       # applies marks: bold, italic, underline, strike, code, link
│   ├── BulletListNode.tsx
│   ├── OrderedListNode.tsx
│   ├── ListItemNode.tsx
│   ├── BlockquoteNode.tsx
│   ├── CodeBlockNode.tsx
│   ├── HardBreakNode.tsx
│   ├── HorizontalRuleNode.tsx
│   ├── ImageNode.tsx
│   └── YoutubeNode.tsx
└── index.ts               # barrel: TipTapRenderer, defaultRegistry, NodeRegistry, TipTapNode
```

## Adding a Custom Content Block

To add a custom TipTap node (e.g., a "hero" full-width banner):

**1. Create the renderer in `packages/ui`**

```tsx
// packages/ui/src/tiptap/nodes/HeroNode.tsx
import type { TipTapNode } from "../types"

interface Props {
  node: TipTapNode
}

export const HeroNode = ({ node }: Props) => (
  <section class="hero">
    <img src={node.attrs?.imageUrl} alt={node.attrs?.title} />
    <h2>{node.attrs?.title}</h2>
    {node.attrs?.buttonText && (
      <a href={node.attrs.buttonHref}>{node.attrs.buttonText}</a>
    )}
  </section>
)
```

**2. Register it**

```ts
// packages/ui/src/tiptap/registry.ts — add one line
import { HeroNode } from "./nodes/HeroNode"

export const defaultRegistry: NodeRegistry = {
  // ... existing entries ...
  hero: HeroNode,
}
```

**3. Add the TipTap extension in `apps/admin`** (separate step — see TipTap extension docs).

No changes to `TipTapRenderer.tsx` or any page file in either app.

## Rebuild Flow

1. Admin saves content → API calls `POST {PUBLIC_CLIENT_URL}/api/revalidate`
2. `revalidate.ts` verifies `x-revalidate-secret` with `crypto.timingSafeEqual`
3. Valid → calls `DEPLOY_HOOK_URL`; absent in dev → returns `{ ok: true }`
4. New build fetches fresh data → regenerates all static HTML

See `docs/rebuild-flow.md` for details.

## How Content Rendering Works

The admin stores page content as **TipTap JSON** — a tree of nodes. Example:

```json
{
  "type": "doc",
  "content": [
    {
      "type": "heading",
      "attrs": { "level": 1 },
      "content": [{ "type": "text", "text": "Hello World" }]
    },
    {
      "type": "paragraph",
      "content": [{ "type": "text", "text": "Welcome." }]
    }
  ]
}
```

`TipTapRenderer.astro` walks this tree recursively. For each node it looks up `node.type` in a **registry** and delegates rendering to the matched Astro component.

```
TipTapRenderer
  └─ looks up node.type in registry
       ├─ "doc"       → Doc.astro       (renders children)
       ├─ "heading"   → Heading.astro   (<h1>–<h6>)
       ├─ "paragraph" → Paragraph.astro (<p>)
       ├─ "text"      → Text.astro      (text + marks)
       ├─ "image"     → Image.astro     (<img>)
       └─ "youtube"   → Youtube.astro   (<iframe>)
```

If a node type is not in the registry, it renders a `<span data-unknown-node="type">` — visible in the HTML but non-breaking.

## Adding a Custom Content Block

To add a custom TipTap node (e.g., a "hero" full-width banner):

**1. Create the renderer component**

```astro
---
// src/shared/tiptap/nodes/Hero.astro
interface Props {
  attrs: { title: string; imageUrl: string; buttonText?: string; buttonHref?: string }
}
const { attrs } = Astro.props
---
<section class="hero">
  <img src={attrs.imageUrl} alt={attrs.title} />
  <h2>{attrs.title}</h2>
  {attrs.buttonText && <a href={attrs.buttonHref}>{attrs.buttonText}</a>}
</section>
```

**2. Register it**

```ts
// src/shared/tiptap/registry.ts
import Hero from "./nodes/Hero.astro"

export const defaultRegistry: NodeRegistry = {
  // ... existing entries ...
  hero: Hero,
}
```

That's it. No changes to `TipTapRenderer.astro` or any page file.

**3. Add the extension to the TipTap editor in `apps/admin`** (separate step — see TipTap extension docs).

## Rebuild Flow

When the admin saves content:
1. API server calls `POST {PUBLIC_CLIENT_URL}/api/revalidate` with `x-revalidate-secret` header
2. `apps/web/src/pages/api/revalidate.ts` verifies the secret
3. If valid → calls `DEPLOY_HOOK_URL` (Render.com rebuilds the static site) or returns `{ ok: true }` locally
4. New build fetches fresh data from the API and generates updated HTML

See `docs/rebuild-flow.md` for details.
